11.Proxy代理
  • 为对象设置访问代理器
  • 比Object.defineProperty()更强大
  • 轻松监视到对象读写过程
  • 非侵入的方式监管了对象的读写
  • 结合对象处理静态类Reflect使用
  • 如果没有对proxy添加方法成员，内部的set,get方法实质也是调用了Reflect对应方法
//对象的监视
const person = {
    name:'wangbo',
    age:24,
    job:'打工人'
}
const personProxy = new Proxy(person,{
    get(target,property){
        console.log(target,property)//{ name: 'wangbo', age: '24' } name
        // return 100
        return target[property]?target[property]:undefined
    },
    set(target,property,value){
        if(property === 'age'){
            if(!Number.isInteger(value)){
                throw new TypeError(`${value} is not an int`)
            }
        }
        target[property] = value
    },
    deleteProperty(target,property){
        console.log('delete',property)//delete job
        delete target[property]
    }
})
console.log(personProxy.name,personProxy.haha)
//wangbo undefined
personProxy.age = 99
console.log(personProxy.age)
//99
// personProxy.age = '88'
// console.log(personProxy.age)
//TypeError: 88 is not an int
delete personProxy[job]
console.log(personProxy)
//{ name: 'wangbo', age: 99 }

//数组的监视

17.Symbol() 数据结构 :一种全新的原始数据类型

  • 场景：
    1.引入多个文件，但是其中有相同的命名
    2.目前项目中大量使用第三方模块，我们可能会尝试扩展模块功能，但可能会定义相同属性名破坏掉原始模块的功能

  • 以前的解决方式：
    1.写代码时约定，在名字前面加上区别标识：如a_name,b_name

  • 缺陷：只是规避了问题，并未解决问题

  • 解决：Symbol(); 为对象添加独一无二的属性名
    1.typeof(Symbol()) //symbol
    2.通过Symbol()创建的每个值都是独一无二，不会相等
    3.Symbol('name'),可以传字符串值让我们区别Symbol(),但是如果传入同样的值两个Symbol仍然不相等
    4.可以作为对象类型属性值[es2015]新增
    5.实现正真的对象私有成员

20.迭代器设计模式
//开发一个任务清单应用
const todos = {
    life:['吃饭','睡觉','打豆豆'],
    learn:['语文','数学','外语'],
    work:['摸鱼']
}
//让协同者不必关心你代码内部的数据结构，向外暴露一个遍历的接口
const todos = {
    life:['吃饭','睡觉','打豆豆'],
    learn:['语文','数学','外语'],
    work:['摸鱼'],
    //原始的方式
    each:function(callback){
        const all = [].concat(this.life,this.learn,this.work)
        for(let item of all){
            callback(item)
        }
    },
    //使用迭代器的方式
    [Symbol.iterator]:function(callback){
        const all = [...this.life,...this.learn,...this.work]
        let index = 0
        return {
            next(){
                return {
                    value:all[index],
                    done:index++>=all.length
                }
            }
        }
        
    }
}
for(let item of todos){
    console.log(item)
}
//吃饭 睡觉 打豆豆 语文 数学 外语 摸鱼
21.生成器函数：Generator
  • 生成器函数返回一个生成器对象
  • 这个对象具有一个next()方法,调用这个方法函数体才会开始执行
  • 生成器函数也实现了iterator接口
  • 惰性执行，遇到yield关键字就停，重新next()继续执行，直到done:true
function * gen(){
    console.log(111)
    yield 100;
    console.log(222)
    yield 200
    console.log(333)
    yield 300
}
const generator = gen()
console.log(generator.next())//111 { value: 100, done: false }
console.log(generator.next())//222 { value: 200, done: false }
console.log(generator.next())//333 { value: 300, done: false }
console.log(generator.next())//{ value: undefined, done: true }

//发号器实现
function * postNumber(){
    let index = 0;
    while(true){
        yield index++
    }
}
const postNum = postNumber()
console.log(postNum.next().value)//0
console.log(postNum.next().value)//1
console.log(postNum.next().value)//2
console.log(postNum.next().value)//3
//生成器实现iterator
const todos = {
    life:['吃饭','睡觉','打豆豆'],
    learn:['语文','数学','外语'],
    work:['摸鱼'],
    //使用生成器的方式
    [Symbol.iterator]:function *(){
        const all = [...this.life,...this.learn,...this.work]
        for(let item of all){
            yield item
        }    
    }
}
for(let item of todos){
    console.log(item,'生成器')
}

二、解决javascript类型系统的问题

  • 大大提高代码的可靠程度

  • 强类型与弱类型（类型安全）
    强类型：限制形参和实参的类型一致，不允许任意的隐式类型转化；弱类型：不限制实参的类型，允许任意的隐式类型转化

  • 静态类型与动态类型（类型检查）：是否允许随时修改变量类型

  • javascript类型系统特征：弱类型、动态类型；几乎没有类型限制，但是丢失了可靠性

  • 弱类型产生的问题
    1.没有编译环节，缺失了编译环节的类型检查。错误得等到运行时才能发现
    2.类型不明确导致函数功能发生改变：如：100 + '100'
    3.对对象索引器赋值，key都会转变为字符串

  • 强类型的优势
    1.错误可以更早暴露，因为在编译阶段做了类型检查
    2.代码更智能，编码更准确
    3.重构会更牢靠
    4.在代码层面减少不必要的类型判断

  • 核心思想：设置引用数，判断当前引用数是否为0
  • 引用计数器（运行效率略慢）
  • 引用关系改变时修改引用数字
  • 引用数字为0时立即回收

    1. 发现垃圾时立即回收（引用计数为0则是垃圾）
    2. 最大限度减少程序暂停（内存爆满时，引用计数算法会找到引用计数值为0的对象释放其空间，保证内存不会占满，减少程序暂停）

    2. 时间开销大[当前引用计数算法需要维护对象引用数值的变化，越多越消耗]
5.GC(标记清除算法)
  • 核心思想：分标记和清楚两个阶段
  • 遍历所有对象找到活动对象（可达对象）并标记（会递归查找可达对象）
  • 遍历所有对象清楚没有标记对象，并清除对象身上的标记以便于第二次开启算法
  • 回收相应的空间
  • 优点：
    1. 可以解决对象循环引用的对象的回收【相对于引用计数】，（因为他在局部作用域，函数执行完后是不可达的）
  • 缺点：
    1. 不能立即回收垃圾对象
    2. 容易产生碎片空间：（未被标记的对象很可能地址不连续，回收的空间也是不连续的）图示如下
6.GC(标记整理算法)
  • 标记整理可以看作是标记清除的增强
  • 标记阶段的操作和标记清除一致
  • 清除阶段会先进行整理，把所有活动对象移动到一起，直接回收右边的空间
  • 优点：解决了碎片空间的问题

7.JavaScript执行引擎：V8
  • 采用了优秀的内存管理机制
  • 采用即时编译（直接将源码翻译成机器码执行，不用再转换成字节码）
  • V8内存设限（上限设置）：bit64:1.5G|bit32:800M,原因如下：
    1. 运行网页足够使用
    2. 垃圾回收机制：官方测试:当垃圾内存达到1.5G时，采用增量标记算法进行垃圾回收只需50ms，而采用非增量标记算法需要1s,从用户体验讲，1s钟是比较长的。
1.V8垃圾回收策略：
  • 采用分代回收的思想
  • 内存分为新生代、老生代
  • 针对不同对象采用不同算法（使垃圾回收更高效）
  • 常用的GC算法：分代回收、空间复制、标记清除、标记整理、标记增量
2.V8内存分配：
  • V8内存空间一分为二
  • 小空间用于存储新生代对象(bit64 32M | bit32 16M)
  • 新生代指存活时间较短的对象（如局部变量在代码块执行完被回收，存活时间较短）
3.新生代对象回收实现：
  • 采用复制算法 + 标记整理
  • 新生代内存区分为两个等大小空间，如上面的from和to
  • 使用空间为From,空闲空间为To（平时不使用）
  • 活动对象存储于From空间，一旦From空间应用到一定程度就会触发GC操作
  • GC操作（采用标记整理(找标记->整理)将活动对象拷贝至To）
  • 把From空间完全释放和To进行交换，之前的From变成了To,之前的To又变成了From
4.回收细节说明：
  • 拷贝过程中可能存在晋升（将新生代移动至老生代）
  • 拷贝过程中如果发现某一个变量对象在老生代对象也存在，那么晋升
  • 一轮GC还存活的新生代需要晋升
  • To空间的使用率超过25%，需要晋升。如果不晋升可能会导致空间不够用
5.老生代对象回收实现：
  • 存放在右侧老生代区域
  • 空间大小（bit64 1.4G | bit32 700M）
  • 老生代对象指存活时间较长的对象，如全局变量、闭包等
  • 采用标记清除、标记整理、增量标记算法
  • 首先使用标记清除完成垃圾空间的回收（空间碎片）【主要的】
  • 如果发现晋升时老生代存储空间无法存储晋升的对象，那么触发标记整理进行空间优化
  • 采用增量标记进行效率优化（垃圾回收会阻塞程序执行，如下图所示，对垃圾进行分段回收【直接可达对象|间接可达对象】）
6.回收细节对比
  • 新生代垃圾回收采用空间换时间（新生代空间小，牺牲to空间是微不足道的，对于时间上的提升是显著的）
  • 老生代不适合复制算法（空间较大，牺牲一半较为浪费，其次，存储的对象较多，牺牲时间去复制是不划算的）
8.Performance工具
  • GC的目的是为实现内存空间的良性循环
  • 合理使用内存是我们所需要做到的
  • Performance提供多种监控方式（内存等）
  • 使用步骤：1.输入目标地址（不跳转）->F12(performance)->开启录制功能,跳转页面->执行用户行为（点击等）->一段时间后停止录制->分析界面中的内存信息
9.内存问题的外在表现[网络良好情况下]
  • 页面延迟加载、经常性暂停（可能频繁的垃圾回收，有代码让内存爆掉）
  • 页面持续性出现糟糕的性能 (内存膨胀，为达到性能申请了远超设备的内存大小)
  • 页面性能随着时间延长越来越差（某些代码导致内存泄漏，内存使用持续升高）
