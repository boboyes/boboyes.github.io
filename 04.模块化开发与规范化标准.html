模块化开发与规范化标准

一. 模块化开发

  • 当下最重要的前端开发范式
  • 随着前端应用日益复杂，将不同类型功能的代码划分不同的模块，单独维护，提高开发效率
1. 模块化的演变过程：模块化的方式
  • 文件划分方式:一个文件就是一个独立的模块
//module-a.js
var name = 'module-a'
function method1 () {
    console.log(name + '#method1')
}
function method2 () {
    console.log(name + '#method2')
}

//modules-b.js
var name = 'module-b'
function method1 () {
    console.log(name + '#method1')
}
function method2 () {
    console.log(name + '#method2')
}

//index.html
<script src="module-a.js"></script>
<script src="module-b.js"></script>
<script>
    //命名冲突
    method1()
    //模块成员可以被修改
    name = 'foo'
</script>

缺陷：
# 没有私有空间，污染全局所用域
# 命名冲突
# 无法管理模块之间的依赖关系
# 模块化完全依靠约定，项目过大显然不适合

  • 命名空间方式：
//module-a.js
var moduleA = {
    name:'module-a',
    method1:function(){
        console.log(this.name+ 'method1')
    },
    method2:function(){
        console.log(this.name+ 'method2')
    }
}

//modules-b.js
var moduleB = {
    name:'module-b',
    method1:function(){
        console.log(this.name+ 'method1')
    },
    method2:function(){
        console.log(this.name+ 'method2')
    }
}

//index.html
<script src="module-a.js"></script>
<script src="module-b.js"></script>
<script>
    //解决了命名冲突
    moduleA.method1()
    moduleB.method1()
    //模块成员仍然可以被修改
    moduleA.name = 'foo'
</script>

缺陷：
# 没有私有空间，污染全局所用域
# 无法管理模块之间的依赖关系

  • IIFE立即执行函数：可在立即函数中传参作为依赖声明，模块之间的关系变得明显
//module-a.js
;(function () {
    var name = 'module-a'
    function method1 () {
        console.log(name + '#method1')
    }
    function method2 () {
        console.log(name + '#method2')
    }
    window.moduleA = {
        method1,
        method2
    }
})()


//modules-b.js
;(function () {
    var name = 'module-b'
    function method1 () {
        console.log(name + '#method1')
    }
    function method2 () {
        console.log(name + '#method2')
    }
    window.moduleB = {
        method1,
        method2
    }
})()

//index.html
<script src="module-a.js"></script>
<script src="module-b.js"></script>
<script>
    //解决了命名冲突
    moduleA.method1()
    moduleB.method1()
    //模块私有成员无法访问
    console.log(moduleA.name)
</script>

缺陷：
以上是模块化的原始方式，通过约定完成模块化的代码组织，不同开发者实施有不同的差别
通过script方式，模块加载不受代码控制，时间长了容易忘记

2. 模块化规范
  • 模块化的原始方式，通过约定完成模块化的代码组织，不同开发者实施有不同的差别。所以急需一套统一的规范
  • 模块化标准 + 模块加载器
1.CommonJS规范：Nodejs提出的一套模块化标准
  • 一个文件就是一个模块
  • 每个模块都有单独的作用域
  • 通过module.exports导出成员
  • 通过require函数载入模块
  • 对比：
    1. CommonJS是以同步的模式加载模块，node的执行机制是启动时加载模块，执行过程中不需要加载，只需要使用
    2. 如果在浏览器端仍然使用此规范，每一次页面加载，都会有大量的同步模式请求，导致效率低下
2.AMD(Asynchronous Module Definition)异步模块定义
  • 绝大多数库都支持CMD规范
  • AMD使用相对复杂
  • 模块JS文件请求频繁
//Require.js

//定义一个模块，必须通过define函数
define('module1',['jquery','./module2'],function($,module2){
    return {//导出成员
        start:function(){
            $('body').animate({margin:'200px'})
            module2()
        }
    }
})

//载入模块:require会自动创建script标签发送对应请求，执行获取的代码
require(['./module1',function(module1){
    module1.start()
}])
3.Sea.js + CMD 通用模块规范定义【淘宝团队推出】
//CMD规范（类似CommonJS规范）
define(function(require,exports,module){
    var $ = require('jquery')

    module.exports = function() {
        console.log('module 2')
        $('body').append('<p>module2</p>')
    }
})

  • ES module 是ES6中定义的模块系统
  • 标准推出时所有主流浏览器不支持此规范，随着webpack等一系列打包工具的流行，这一规范开始普及

    1. 导入和导出
    • 导入导出的语法并非解构，而是模块的特定语法
    • 导出的成员是引用，并非复制一份
    • 导出的成员是只读的（常量）
        //app.js
    import {name} from './module-a.js'
    import {hello,person} from './module-b.js' /*此处语法也并非解构，而是固定语法*/
    import {default as Person} from './module-c.js'
    import nameDefault from './module-d.js'//可任意命名
    console.log(name)

    //module-a.js
    export var name = 'bobo'
    export function hello(){}
    export class Person{}

    //module-b.js
    var name = 'bobo'
    function hello(){}
    class Person{}
    export {name,hello,Person} 
    //注意：此处导出的不是对象字面量，这只是一个固定语法
    //导出对象的方式：export default {name,hello,Person}

    //module-c.js
    var name = 'bobo'
    function hello(){}
    class Person{}
    export {
        name as fooName,//重命名
        hello as fooHello,
        Person as default //default是默认导出成员
    }

    //module-d.js
    var nameDefault = 'bobo'
    export default nameDefault

3. 常用的模块化打包工具
  1. 由来
  • ES Module存在兼容性问题
  • 模块文件过多，网络请求频繁
  • 所有前端资源都需要模块化
  1. 功能
  • 编译代码[转换新特性]
  • 把模块化代码再打包到一起，减少请求
  • 支持模块化打包所有类型前端资源

  1. 主流模块化打包工具：webpack, parcel, rollup

  2. webpack 模块打包器（Module bundler）【前端模块化】

  • 模块加载器 (loader)
  • 代码拆分 (Code Splitting)
  • 资源模块 (Asset Module)【加载多种类型资源】

  • 打包结果运行原理：使用打包模式为'none'的结果去控制台调试分析

  • 运行特点
    0. webpack只是打包工具，编译转换依靠加载器loader

    1. 只能打包js,其他的类型打包需要使用loader
    2. 建议根据代码的需要动态导入资源，需要资源的是代码，不是应用
    3. 因为模块打包需要，所以去处理了import和export,不能去转换其他的es6特性，需要其他的插件
    4. 支持import require().default define AMD[require],尽量同项目使用同标准

  • html文件中支持的两种加载方式

    1. img:src
    2. a:href //需要手动配置，默认不识别

  • 样式文件中支持的两种加载方式

    1. background:url()
    2. @import url(reset.css)

  • 加载器类型

    1. 编译转换类：css-loader
    2. 文件操作类：将加载的模块拷贝到输出目录，同时将文件的访问路径导出：file-loader
    3. 代码检查类: 校验代码、统一代码风格，提高代码质量

  • 开发一个loader:一个markdown文件加载器

    1. 每个webpack-loader需要导出一个函数，就是对所加载的资源的处理过程
    2. 输入是所加载的资源文件内容，输出就是加工过后的结果
    3. 输出的最终结果必须是一段javascript代码
    4. 开发的loader可以不输出js代码，那么就需要使用额外的loader帮助解析
        const marked = require('marked') //md解析模块
    module.exports = source => {
        //return 'console.log("hello")'

        //return `module.exports = "${html}"`  //此处如果直接导出会省略换行符
        return `module.exports = ${JSON.stringify(html)}`//包含换行符信息
        return `export default ${JSON.stringify(html)}`//esm方式导出
    }

  • webpack插件-通过钩子机制实现

    1. 相比与Loader，Plugin拥有更宽的能力范围,触及webpack的每一个环节
    2. webpack工作有很多环节，为便于扩展，webpack给每一个环节都埋下了钩子
    3. 查看官方文档，查看webpack插件的钩子
    4. webpack插件必须是一个函数或者是一个包含apply方法的对象
    5. webpack插件会在被调用时直接去调用内部的apply方法
        class MyPlugin {
        apply(compiler){//接受一个compiler对象参数，webpack工作过程中最核心的一个对象，包含了此次构建的所有配置信息
            
            //明确任务执行的时机，及找到对应的钩子emit
            compiler.hooks.emit.tab('MyPlugin',compilation => {//第一个参数是插件的名称
                //compilation可理解为本次打包的上下文
                for(const name in compilation.assets){
                    //每一项就是被处理文件的名称console.log()
                    //console.log(name,compilation.assets[name].source())//打印文件名和文件内容
                    if(name.endsWidth('.js')){
                        const contents = compilation.assets[name].source()
                        const withoutComments = contents.replace(/\/\*\*+\*\//g,'')//匹配所有开头注释
                        compilation.assets[name] = {
                            source:()=> withoutComments,
                            size:() => withoutComments.length //webpack要求的必须的方法
                        }
                    }
                }
            })
        }
    }

        //开启HMR:
    1. yarn webpack-dev-server --hot
    2. 配置文件开启: 查看前面的代码配置

    //webpack中的HMR 需要手动处理模块热替换逻辑，简单配置只能实现css热更新
    //除css外的其他代码，往往是没有规律的
    //某些框架的文件都是有规律的，所以我们在使用框架时发现可以热更新，是因为框架做了热更新处理

    //使用HMR APIs  hot是核心API
    //手动热替换替代浏览器自动刷新：拦截了监听操作
    module.hot.accept('./editor',()=>{
        console.log('editor模块被我拦截了，请自己设置刷新动作替换自动刷新')
    })

    //示例，实现热更新【主要代码】
    //editor.js
    import './editor.css'

    export default () => {
        const editorElement = document.createElement('div')
        editorElement.contentEditable = true
        editorElement.className = 'editor'
        console.log('editor init completed')
        alert('12')
        return editorElement
    }
    
    //main.js
    import createEditor from './editor'
    import background from './better.png'
    import './global.css'
    const editor = createEditor()
    document.body.appendChild(editor)
    const img = new Image()
    img.src = background
    document.body.appendChild(img)
    // ================================================================
    // HMR 手动处理模块热更新
    // 不用担心这些代码在生产环境冗余的问题，因为通过 webpack 打包后，
    // 这些代码全部会被移除，这些只是开发阶段用到
    if (module.hot) {
        let hotEditor = editor
        module.hot.accept('./editor.js', () => {
            // 当 editor.js 更新，自动执行此函数
            // 临时记录编辑器内容
            const value = hotEditor.innerHTML
            // 移除更新前的元素
            document.body.removeChild(hotEditor)
            // 创建新的编辑器
            // 此时 createEditor 已经是更新过后的函数了
            hotEditor = createEditor()
            // 还原编辑器内容
            hotEditor.innerHTML = value
            // 追加到页面
            document.body.appendChild(hotEditor)
        })

        module.hot.accept('./better.png', () => {
            // 当 better.png 更新后执行
            // 重写设置 src 会触发图片元素重新加载，从而局部更新图片
            img.src = background
        })
    // style-loader 内部自动处理更新样式，所以不需要手动处理样式模块
    }

    //HMR注意事项
    1. 处理代码有错误也会自动刷新，导致页面错误被清除
        将 {hot:true} -> {hotOnly:true}
    2. 使用HMR需配置：new webpack.HotModuleReplacementPlugin()  //因为module.hot由此插件提供
    3. 与业务功能无关的热替换代码会被移除，不必担心会打包到生产环境
